
# Git 与 GitHub 常见操作 Q&A

### Q1: GitHub 仓库如何克隆？

**A:**  
克隆 GitHub 仓库到本地需要使用 `git clone` 命令。

1. **获取仓库地址**  
   在 GitHub 仓库页面点击绿色的 `Code` 按钮，复制 HTTPS 或 SSH 地址。

2. **执行克隆命令**  
   打开终端（命令行），进入希望存放项目的目录，运行：
   ```bash
   git clone <仓库地址>
   ```
   例如：
   ```bash
   git clone https://github.com/用户名/仓库名.git
   ```

3. **进入项目目录**  
   ```bash
   cd 仓库名
   ```
   现在你就可以在本地查看和编辑代码了。

### Q2: 如何进行代码版本的回溯？

**A:**  
代码版本回溯根据场景不同有三种主要方式：**临时查看**、**安全撤销（revert）** 和 **强制重置（reset）**。

#### 方式一：临时查看历史版本（不影响分支）
- **命令**：`git checkout <commit-id>`
- **效果**：
  - 工作区内容变为指定提交时的模样，你可以浏览、测试、甚至基于此版本创建新分支。
  - 分支指针（如 `main`）**并未移动**，历史记录完全不变。
  - 终端会提示你处于 "detached HEAD" 状态。
- **返回最新状态**：执行 `git switch <原分支名>` 或 `git checkout <原分支名>` 即可。

#### 方式二：安全回溯（推荐用于已推送的协作分支）
使用 `git revert` 创建一个**反向提交**来抵消之前的错误修改。
- **命令**：
  ```bash
  git revert <commit-id>
  ```
- **效果**：
  - 生成一个**新的提交**，其内容恰好是目标提交的“逆操作”（例如目标提交新增了一行，revert 就删除那一行）。
  - 分支历史**向前推进**，不会丢失任何历史记录，所有人都能清楚地看到“撤销某次提交”的操作记录。
  - 可以直接 `git push` 到远程，无需强制推送，不影响协作者的本地仓库。
- **适用场景**：公共分支（如 `main`/`develop`）上已经推送的错误提交，需要安全地撤销功能。

#### 方式三：强制回溯（修改本地/私有分支历史）
使用 `git reset` 直接移动分支指针，丢弃后续提交。
- **命令**：
  ```bash
  git reset --hard <commit-id>
  ```
- **效果**：
  - 分支指针直接**回退**到指定提交，之后的所有提交记录被**彻底删除**（但可通过 `reflog` 找回一段时间）。
  - 工作区和暂存区也同时强制变成目标版本的状态，所有未提交的修改都会丢失。
  - 若该分支已推送至远程，需要 `git push -f` 强制覆盖，**队友 pull 时会遇到冲突，需要强制同步本地**。
- **适用场景**：本地尚未推送的错误提交，或你明确知晓只有自己使用的私有分支。

#### 辅助命令
- 查看提交历史：`git log --oneline`
- 查看所有操作记录（含 `reset` 丢失的提交）：`git reflog`

### Q3: 如何拉取最新代码？

**A:**  
拉取最新代码即将远程仓库的更新同步到本地，使用 `git pull` 命令。

#### 基本操作
在项目目录下执行：
```bash
git pull
```
- **效果**：
  - 从远程仓库下载最新提交，并与当前本地分支**合并**。
  - 若远程有更新且无冲突，本地分支指针会向前移动，工作区自动更新。
  - 若存在冲突，Git 会提示你手动解决冲突后再提交。

#### 详细说明
- `git pull` = `git fetch` + `git merge`
  - `git fetch origin`：获取远程所有更新，但不自动合并到本地分支，工作区**不变**。
  - `git merge origin/<分支名>`：将远程分支的更新合并到当前本地分支，**工作区会更新为合并后的内容**。

#### 避免合并冲突的建议
- 拉取前确保本地工作区是干净的（`git status` 无未提交修改）。
- 若有未提交修改，可先 `git stash` 暂存，拉取后再 `git stash pop` 恢复。
- 使用 `git pull --rebase` 可保持提交历史线性整洁（需谨慎处理冲突）。
