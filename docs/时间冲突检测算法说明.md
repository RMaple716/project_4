## 1. 功能概述

时间冲突检测算法是行程规划系统的核心校验模块，用于确保生成的行程安排合理、无时间冲突。该算法能够：

- ✅ 检测活动时间重叠
- ✅ 验证游览时长合理性
- ✅ 检查时间安排是否符合日常作息
- ✅ 控制每日行程总时长
- ✅ 提供优化建议

## 2. 核心算法设计

### 2.1 时间区间重叠检测算法

**算法原理**: 使用经典的区间相交判断

```python
def check_time_overlap(start1, end1, start2, end2):
    """
    两个区间不重叠的条件: end1 <= start2 OR end2 <= start1
    因此重叠的条件是: start1 < end2 AND start2 < end1
    """
    return start1 < end2 and start2 < end1
```

**图示说明**:
```
情况1: 有重叠
活动A: |--------|
活动B:     |--------|
          ↑ 重叠区域

情况2: 无重叠
活动A: |----|
活动B:         |----|
```

### 2.2 时间解析算法

支持多种时间格式输入：

| 输入格式 | 示例 | 解析结果（分钟） |
|---------|------|----------------|
| HH:MM | "09:30" | 570 |
| 时间槽（英文） | "morning" | 540 (9:00) |
| 时间槽（中文） | "上午" | 540 (9:00) |
| 默认值 | - | 540 (9:00) |

**时间槽映射表**:
```python
time_slots = {
    "morning": 540,    # 9:00
    "afternoon": 840,  # 14:00
    "evening": 1080,   # 18:00
    "上午": 540,
    "中午": 720,
    "下午": 840,
    "晚上": 1080
}
```

### 2.3 时长解析算法

支持灵活的时长描述：

| 输入格式 | 示例 | 解析结果（分钟） |
|---------|------|----------------|
| X小时 | "2小时" | 120 |
| X-Y小时 | "2-3小时" | 150 (平均值) |
| X分钟 | "30分钟" | 30 |
| 特殊词汇 | "半天" | 240 |
| 特殊词汇 | "全天" | 480 |

## 3. 冲突检测类型

### 3.1 时间重叠冲突 (time_overlap)

**严重程度**: `error`

**检测逻辑**: 两两比较所有活动的时间区间

**示例**:
```json
{
  "type": "time_overlap",
  "description": "'故宫博物院' (09:00-12:00) 与 '午餐' (11:30-12:30) 时间重叠",
  "severity": "error",
  "activities": ["故宫博物院", "午餐"]
}
```

### 3.2 不合理时间 (unreasonable_time)

**严重程度**: `warning`

**检测规则**:
- 开始时间早于 6:00 (360分钟)
- 结束时间晚于 23:00 (1380分钟)

**示例**:
```json
{
  "type": "unreasonable_time",
  "description": "'凌晨活动' 开始时间过早 (03:00)",
  "severity": "warning",
  "activities": ["凌晨活动"]
}
```

### 3.3 游览时长过短 (too_short_duration)

**严重程度**: `warning`

**检测规则**: 景点游览时间 < 30分钟

**示例**:
```json
{
  "type": "too_short_duration",
  "description": "'快速参观' 游览时间过短 (20分钟)，建议至少1小时",
  "severity": "warning",
  "activities": ["快速参观"]
}
```

### 3.4 游览时长过长 (too_long_duration)

**严重程度**: `warning`

**检测规则**: 景点游览时间 > 480分钟 (8小时)

**示例**:
```json
{
  "type": "too_long_duration",
  "description": "'深度游' 游览时间过长 (600分钟)，建议分多天游览",
  "severity": "warning",
  "activities": ["深度游"]
}
```

### 3.5 过度安排 (overloaded_day)

**严重程度**: `warning`

**检测规则**: 单日行程总时长 > 12小时

**示例**:
```json
{
  "type": "overloaded_day",
  "description": "当日行程总时长 15.0 小时，建议不超过12小时",
  "severity": "warning",
  "activities": ["活动1", "活动2", "活动3"]
}
```

### 3.6 预算超支 (budget_exceeded)

**严重程度**: `error`

**检测规则**: 实际花费 > 用户预算

**示例**:
```json
{
  "type": "budget_exceeded",
  "description": "总花费 5500 元超出预算 5000 元",
  "severity": "error",
  "activities": []
}
```

## 5. 使用示例

### 5.1 Python 调用示例

```python
import requests

# 构造测试数据
schedule = [
    {
        "name": "故宫博物院",
        "start_time": "09:00",
        "end_time": "12:00",
        "activity_type": "attraction"
    },
    {
        "name": "午餐",
        "start_time": "12:30",
        "duration": "1小时",
        "activity_type": "meal"
    }
]

# 调用API
response = requests.post(
    "http://127.0.0.1:8000/api/v1/validation/time-conflict",
    json={"schedule": schedule}
)

result = response.json()
if result["data"]["has_conflict"]:
    print("发现时间冲突:")
    for conflict in result["data"]["conflicts"]:
        print(f"  - {conflict['description']}")
else:
    print("行程安排合理，无时间冲突")
```

### 5.2 前端集成示例

```javascript
// 在生成行程后自动校验
async function validateItinerary(dayPlans) {
  const response = await fetch('/api/v1/validation/itinerary', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ day_plans: dayPlans })
  });
  
  const result = await response.json();
  
  if (!result.data.valid) {
    // 显示冲突警告
    result.data.conflicts.forEach(conflict => {
      showWarning(conflict.description, conflict.severity);
    });
  } else {
    showMessage('行程校验通过！');
  }
  
  return result.data;
}
```

## 7. 算法优化建议

### 7.1 当前限制

1. **交通时间估算简化**: 目前使用固定值（15-30分钟），未考虑真实距离
2. **单天检测**: 暂未实现跨天的连续性检查
3. **开放时间校验**: 未对接景点真实的开放时间数据

### 7.2 未来改进方向

1. **集成地图API**: 使用高德/百度地图API获取真实交通时间
2. **智能缓冲**: 根据活动类型自动添加合理的休息时间
3. **动态调整**: 检测到冲突后自动推荐调整方案
4. **多日优化**: 考虑多日行程的均衡性，避免某天过累

## 9. 常见问题 FAQ

### Q1: 为什么使用时间槽而不是精确时间？
**A**: 时间槽（上午/下午/晚上）降低了算法复杂度，避免了处理"09:30-11:30"这种具体时间冲突的难度。后续可以根据实际需求升级为精确时间。

### Q2: 如何处理跨天的活动？
**A**: 当前版本假设每个活动都在同一天内完成。对于跨天活动（如长途火车），建议在交通智能体中单独处理，不在时间冲突检测范围内。

### Q3: 冲突检测的准确率如何？
**A**: 对于明确指定时间的活动，准确率为100%。对于使用时间槽的活动，可能存在误判（因为时间槽代表的是一个时间段而非精确时间点）。

### Q4: 能否自动修复冲突？
**A**: 当前版本仅检测冲突并提供建议，不自动修复。自动修复功能将在后续版本中实现。
