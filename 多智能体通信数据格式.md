这份文档重点定义了成员 B（协调层）与成员 C（智能体层）之间交互的 JSON 格式，确保了各模块在开发时的独立性。

### 📄 多智能体通信数据格式规范 (v1.0)

```markdown
# 多智能体通信数据格式规范 (Multi-Agent Communication Protocol)

> **版本**: v1.0  
> **负责人**: 成员B (需求与协调负责人)  
> **适用对象**: 成员C (智能体开发), 成员D (后端集成)

## 1. 设计原则
1. **统一信封 (Unified Envelope)**: 所有智能体的输入和输出必须包裹在统一的 JSON 结构中，便于协调层解析。
2. **弱耦合 (Loose Coupling)**: 智能体只关心自己的输入参数，不感知其他智能体的存在。
3. **容错性 (Fault Tolerance)**: 即使某个智能体返回空结果，也不应导致整个行程规划崩溃。

---

## 2. 通用消息信封 (Message Envelope)

### 2.1 请求信封 (Request Envelope)
协调层向智能体发送任务时使用的格式：

```json
{
  "task_id": "subtask_001",           // 子任务唯一标识
  "agent_type": "attraction",         // 目标智能体类型: attraction/accommodation/food/transport
  "parameters": {                     // 具体的业务参数
    "city_name": "北京",
    "travel_days": 3,
    "budget": 1000.0
  },
  "timestamp": "2026-05-19T10:00:00Z" // 任务下发时间
}
```

### 2.2 响应信封 (Response Envelope)
智能体完成任务后返回的格式：

```json
{
  "task_id": "subtask_001",           // 对应的子任务ID
  "status": "success",                // 状态: success / failed / partial
  "data": {                           // 核心业务数据
    "items": []                       // 推荐列表
  },
  "metadata": {                       // 元数据
    "processing_time_ms": 150,        // 处理耗时
    "source": "mock_data"             // 数据来源说明
  },
  "error_message": null               // 失败时的错误描述
}
```

## 3. 各智能体详细数据定义

### 3.1 景点智能体 (Attraction Agent)
**职责**: 推荐每日游览地点及建议时长。

#### 输入参数 (`parameters`)
| 字段 | 类型 | 说明 |
|------|------|------|
| city_name | string | 目的地城市 |
| travel_days | int | 计划天数 |
| preferences | array | 兴趣标签 (如 ["历史", "自然"]) |
| dislikes | array | 排除标签 |
| ticket_budget | float | 门票总预算 |

#### 输出数据 (`data.items` 元素结构)
```json
{
  "id": "att_001",
  "name": "故宫博物院",
  "category": "历史古迹",
  "suggested_duration": "4小时",       // 建议游览时长
  "visit_time_slot": "morning",        // 建议时段: morning/afternoon/evening
  "ticket_price": 60.0,
  "location": {                        // 坐标信息，用于路线优化
    "lat": 39.916,
    "lng": 116.397
  },
  "tags": ["必去", "室内"]
}
```

### 3.2 住宿智能体 (Accommodation Agent)
**职责**: 推荐全程或分段的住宿地点。

#### 输入参数 (`parameters`)
| 字段 | 类型 | 说明 |
|------|------|------|
| city_name | string | 目的地城市 |
| check_in_date | string | 入住日期 |
| nights | int | 住宿晚数 |
| budget_per_night | float | 每晚预算上限 |
| location_preference | string | 位置偏好 (如 "市中心", "靠近地铁") |

#### 输出数据 (`data.items` 元素结构)
```json
{
  "id": "hotel_001",
  "name": "北京王府井希尔顿酒店",
  "address": "东城区王府井大街8号",
  "price_per_night": 800.0,
  "rating": 4.8,
  "distance_to_center_km": 1.5,
  "amenities": ["早餐", "WiFi", "停车场"]
}
```

### 3.3 美食智能体 (Food Agent)
**职责**: 推荐当地特色餐厅。

#### 输入参数 (`parameters`)
| 字段 | 类型 | 说明 |
|------|------|------|
| city_name | string | 目的地城市 |
| meal_type | string | 餐别: breakfast/lunch/dinner |
| budget_per_person | float | 人均预算 |
| cuisine_preference | string | 菜系偏好 |

#### 输出数据 (`data.items` 元素结构)
```json
{
  "id": "rest_001",
  "name": "四季民福烤鸭店",
  "cuisine": "京菜",
  "avg_price_per_person": 150.0,
  "rating": 4.7,
  "recommended_dishes": ["酥香嫩烤鸭", "贝勒烤肉"],
  "location": {
    "lat": 39.910,
    "lng": 116.400
  }
}
```

### 3.4 交通智能体 (Transport Agent)
**职责**: 计算景点间或城市间的移动方案。

#### 输入参数 (`parameters`)
| 字段 | 类型 | 说明 |
|------|------|------|
| from_location | object | 起点 {name, lat, lng} |
| to_location | object | 终点 {name, lat, lng} |
| mode_preference | string | 偏好方式: walking/transit/driving |

#### 输出数据 (`data.items` 元素结构)
```json
{
  "from": "故宫博物院",
  "to": "颐和园",
  "mode": "transit",                 // 交通方式
  "duration_minutes": 50,            // 预计耗时(分钟)
  "cost": 5.0,                       // 费用
  "steps": [                         // 详细步骤
    {"instruction": "步行至天安门东站", "distance_m": 500},
    {"instruction": "乘坐地铁1号线", "distance_m": 15000}
  ]
}
```

## 4. 协调层整合逻辑 (Integration Logic)

成员 B 负责将上述四个智能体的输出拼接为 `FinalItinerary`。

### 4.1 每日行程单元 (Daily Unit)
```json
{
  "day_index": 1,
  "date": "2026-05-20",
  "weather": "晴",                   // 来自外部天气API
  "schedule": [
    {
      "time": "09:00-12:00",
      "type": "attraction",
      "content": { ...景点数据... }
    },
    {
      "time": "12:00-13:30",
      "type": "lunch",
      "content": { ...美食数据... }
    }
  ],
  "daily_cost": 350.0,               // 当日总花费汇总
  "accommodation": { ...住宿数据... } // 仅在第一天或换酒店时出现
}
```

## 5. 异常处理约定
1. **超时处理**: 若智能体在 5秒内未返回，协调层将其标记为 `failed`，并使用默认推荐数据（由成员 D 提供的基础数据表）。
2. **预算超标**: 若智能体返回的所有项均超出预算，需在 `metadata` 中标记 `over_budget: true`，协调层将尝试降级推荐。
```

### 💡 针对你没考虑到的事情的建议：

1. **ID 的统一性**：
   建议在 `分工.md` 中明确，所有智能体返回的 `id`（如 `att_001`）必须是全局唯一的或者带有前缀，防止在整合时发生 ID 冲突。

2. **地理位置标准化**：
   我在格式中特意加入了 `location: {lat, lng}`。因为成员 B 需要做“路线优化”，如果景点只有名字没有坐标，是无法计算距离和避免折返的。请提醒成员 C 确保输出的数据包含经纬度。

3. **模拟数据的占位符**：
   由于是“极速开发”，成员 C 可能无法接入真实的地图 API。建议在文档中约定：如果无法获取真实数据，可以返回一个包含 `is_mock: true` 字段的 JSON，这样前端展示时可以加一个“模拟数据”的标签，避免用户误解。

4. **时间槽 (Time Slot) 的预定义**：
   为了简化成员 B 的整合算法，我定义了 `visit_time_slot` (morning/afternoon/evening)。这样成员 B 只需要把标记为 `morning` 的景点排在上午，而不需要去处理复杂的“09:30-11:30”这种具体时间冲突，大大降低了开发难度。
